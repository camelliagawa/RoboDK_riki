## 0. TL;DR（最初にこれだけ読む）

包丁の刃元から刃先にかけて研削量が均一にならない問題を、**送り速度の変調**で解決しようとしている。
現行カーブ（刃元50 → 刃先18 mm/sec）は既に適用済みだが、**まだ刃元が刃先の3.1倍多く削れている**。

今回、実測ログの解析で以下が判明した：

1. **除去量の正しい指標は「力の山の面積 ∫F dt」**（ピーク値ではない）
2. **接触時間に約2.5秒の「下駄 T0」があり、これが速度変調の効きを半減させている**
3. **HaL（2面目）は取り付け角度誤差（推定約0.6°）で刃先が砥石に届いていない**
4. **位置調整をせずに速度カーブを校正しても無意味**（セッティング依存の歪んだカーブになる）

**次にやること**: ①位置調整（**←今ここ／難航中**） → ②押し込み試験で左右対称性を検証 → ③面積ベースで速度カーブ更新

> ⚠ **2026-07-21 の重要な進展 → §0.5 を必ず読むこと。** 押し込み試験を実施し、**HaR の剛性は確定・正常**、**HaL は「刃元偏重の傾き」**と判明。ただし **RoboDK のツール姿勢オフセット（rotx/roty/rotz）では heel/tip の偏りを直せない**ことが3軸実験＋幾何で確定した。次は物理座り直し等（§0.5 参照）。

## 0.5 2026-07-21 セッションの結果（押し込み試験・位置調整）★最新★

### やったこと
- **押し込みスイープ試験**（砥石停止・両面・`--force-limit`）を実施：`transl(2.0/2.5/3.0/3.5/4.0,0,0)` の5点。同一取付の **`air.csv`（`transl(12)` で砥石から離隔）** も取得。

### 判明した確定事項
1. **押し込み方向は「逆」**：`transl` 第1引数が**大きいほど砥石から離れる**（＝小さいほど深い）。実測で 2.0mm が最も強く当たり(最大力)、4.0mm が最弱。air は `transl(12)`＝完全離隔。**→ 速度カーブ逆算や k(x) の符号解釈は「実push＝設定値の逆向き」に注意（剛性は |傾き| で扱えば問題ない）。**
2. **HaR の剛性 k(x) は確定・正常**：刃元 9.7 → 刃先 1.5 N/mm（単調・R²>0.99）。§5.1 の目安（刃元8/刃先2.2）と同オーダー。**HaR は素直に整っている。**
3. **HaL は「刃元偏重（傾き）」**：重力除去後の接触力で **HaL の刃元/刃先比＝8.3倍（transl2.0）／7.1倍（transl3.0）** に対し **HaR は 2.9倍／1.8倍**。HaL は刃元が過剰(≈36N)・刃先が浮く(≈4N・ほぼ非接触)。**刃元↑/刃先↓の“シーソー再配分”＝取り付け角度誤差(傾き)の signature**。グリッパ剛性の左右非対称（一様スケール）ではない。
4. **⚠ ツール姿勢オフセットでは heel/tip を直せない（3軸実験＋幾何で確定）**：`rotx0.5 / roty−0.5 / rotz90.5` を試したが HaL比は 7.08→7.26/7.16/7.46 で**全て無反応**。理由＝**LS上、刃長(X)方向にツール姿勢(W/P/R)は一定**なので、姿勢回転は各点まわりの局所回転にしかならず**ストローク間の“深さ勾配”を作れない**。heel を深く/tip を浅くするには**並進の勾配**が要る（＝物理傾き or 参照フレーム/パスの傾き）。**姿勢回転の探索は打ち切り。**
5. **自動ツールの落とし穴**：`stroke_analyze.py --stiffness` は **air同期失敗（残差2〜2.6N）＋HaL面の山検出破綻（19本にならない）** により「剛性非対称→グリッパ/治具」と**誤結論**を出す。**採用しないこと。** 堅牢なのは下記の自作指標。

### 堅牢な解析メソッド（検出が19本に安定しない本データ用）
- **重力はベクトル差分で除去**（air不要）：面内の**下位25% |F| の平均ベクトル**を非接触時の重力とみなし、生ベクトルから引いて `contact|F|` を得る。
- **heel/tip 比**：面内を時間割合で **刃元＝先頭0〜35%／刃先＝末尾65〜100%** に分け、各領域の `contact|F|` の**上位30%平均**を代表値に、その比を取る。**この比が再現性の高い判定指標**（HaR 1.7〜1.8 で安定、感度0.03）。
- 剛性 k(x) は 5点スイープの各ストロークで `ΔF/Δδ`（ベクトル差分なら重力自動キャンセル）。ただし山検出が不安定なので、当面は heel/tip 比で判定するのが確実。

### air.csv の運用メモ
- **同一取付・同じ kenma・非接触**で撮ると同期が良い。`transl(12)` 離隔は工具位置＝姿勢が少しズレて同期残差が2N残る（**ただし上の結論はベクトル差分法なので air 品質に依存しない**）。**次回 air は `transl(3)` 付近で「砥石だけ逃がす」と同期改善。**

### 次の選択肢（§5.2 と重複・こちらが最新）
- **A. 物理的に包丁を座り直す（推奨・根本）**：HaL の食い込み勾配は、包丁がグリッパ内で傾いて咥えられているのが主因の見込み。座り直して**両面を録り直す**。HaL比が HaR(≈1.8)へ寄れば傾きが原因＝解決。寄らなければ「2つの刃(研ぎ角)が左右非対称」＝姿勢では揃わない → Cへ。
- **B. 参照フレーム「Kenma point」を傾ける**：ソフトで物理相当の深さ勾配を作る。ただし**両面共有**なので HaR も動く＋軸探索が要る。Aで直らない時の代替。
- **C. 左右を揃えず各面を個別運用（現実解）**：HaL を**もっと深く押し込んで刃先を接触**させ（今の刃先は≈3N・ほぼ非接触で速度では削れない）、刃元過剰分は**各面別の速度カーブで“除去量”を揃える**。HaR/HaL は元々別プロファイル。押し込み試験の k(x) がそのまま使える。
- **判定はいずれも同じ heel/tip 比**（HaR≈1.8 に寄るか）。**回転オフセットは 0/90/90 に戻すこと。**

## 2. 物理モデル（ここを誤ると全部ずれる）

### 2.1 Preston則の「速度」は砥石周速であって送り速度ではない

```
除去量 ∝ 圧力 × 相対滑り速度 × 時間
```

TORMEK T-8の砥石周速は約1m/sオーダー。送り速度（18〜50mm/s）より**2桁大きい**。
つまり**送り速度を変えてもPreston則の v はほぼ変わらない**。

送り速度が効くのは別経路 ——「刃のある一点が砥石に接触していられる時間（滞留時間）」。

```
局所除去量 Δh(x) ∝ k · F(x) · v_砥石 · τ(x)
                                      τ(x) = 接触幅 / v_送り(x)

v_砥石 と k は定数 ⇒  Δh(x) ∝ ∫F dt  ← 山の面積（力積, N·s）
```

**結論: 山の面積が除去量の代理指標。ピーク値ではない。**
（ピーク値は「高さ」だけ。面積は「高さ×幅」で接触時間の情報も含む）

### 2.2 接触時間には下駄 T0 がある ★最重要★

```
接触時間 w(x) = STROKE_DIST / v(x) + T0
面積     A(x) = Favg(x) · w(x)
```

**実測値（HaR, 19本）: T0 = 2.50 s、ばらつき 2.42〜2.53 s（±0.05s！）**

19本すべてで定数になるので、このモデルは正しい。

**T0の正体**: 押し込み量が3mmなので、接近/後退の10mm動作のうち最初と最後の3mm分は接触したまま引きずっている。この区間は速度変調されない（1000mm/sec指令）。

**T0が引き起こす問題**:
```
速度比    50/18 = 2.78倍
実測の幅比        1.54倍   ← 半分しか伸びていない
```
→ **速度変調の効果がT0で薄められている。これが「速度を2.78倍落としたのに除去量が3.1倍のまま」の原因。**

**T0が生む理論的な下限**:
```
A_floor(x) = Favg(x) · T0    ← 速度を無限大にしてもこれ以上は削れてしまう
刃元の A_floor = 46.7 N·s
```
刃元の面積を刃先（20.9 N·s）まで下げることは**速度変調では原理的に不可能**。
→ 全ストロークを A_target ≈ 47.6 N·s に合わせるしかなく、刃先を 18 → **5.2 mm/sec** まで落とす必要が出る（研削時間 39.6s → 92.2s、**2.33倍**）。

**→ 対策: 接近/後退を −10mm から −20mm 程度に広げて T0 を削るのが最も効く。**

### 2.3 剛性は直列バネで考える（80倍問題の解決）

過去に片持ち梁モデルが刃元/刃先の力比を **80倍** と予測したが、実測は **2.75〜3倍**だった（30倍近い乖離）。

原因は境界条件。正しくは：

```
1/k_total(x) = 1/k_blade(x) + 1/k_setup

k_blade(x) … 刃のたわみ剛性 3EI(x)/L_eff³。E既知（鋼 約205GPa）、I(x)はSTLから計算可
k_setup    … グリッパ＋治具＋ロボット手首の剛性。★包丁の機種が変わっても同じ★
```

- 刃元: `k_blade` が巨大 → `k_total ≈ k_setup` で**頭打ち**
- 刃先: `k_blade` が小さい → `k_total ≈ k_blade`

⇒ 比は80倍ではなく `k_setup / k_blade_tip` に圧縮される。実測3倍から逆算すると **k_setup は刃元剛性の約1/27**。
**つまり刃元側の剛性を決めているのは包丁ではなくグリッパと治具。**

## 3. 実測で判明した事実

解析対象: `force_log_20260717_151800.csv`（速度変調**適用後**のデータ）+ `air.csv`

### 3.1 air.csv の正体と役割 ★必須★

`force_moment_overlay.py` は `TARE_ON_START=True` で**待機姿勢**の零点を取る。
研削中の姿勢では重力成分が固定オフセットとして乗り、J6で180°反転するHaR/HaLでは面ごとに大きく変わる。

```
air.csv 1面目区間: fy = +8.6N（std 0.23、ほぼ一定）
air.csv 2面目区間: fy = +0.09N, fz = +1.24N
差ベクトルの大きさ = 8.565N → |mg| = 4.28N → 換算質量 437g（グリッパ＋包丁として妥当）
```

**air.csv を引かないと2面目の力が過小評価される。必ず引くこと。**
（`plot_force_log.py` の `apply_baseline_persides` が面ごとに個別整列して減算してくれる）

減算後の接触前区間（0〜11s）の残差 = 0.52N。同期ラグ +0.30s。

### 3.2 面積（＝除去量）の実測値

| | 刃元 | 刃先 | **比** |
|---|---|---|---|
| **HaR（1面目）** | 64.8 N·s | 20.9 N·s | **3.10倍** |
| **HaL（2面目）** | 46.3 N·s | 4.0 N·s | **13.91倍** |

**HaR ですらまだ3.1倍のばらつきがある。速度カーブは全く足りていない。**

### 3.3 HaL の異常 ★位置調整が必要な根拠★

```
T0:  HaR = 2.50s（ばらつき ±0.05s）   ← モデル通り
     HaL = 0.84s（ばらつき -1.38〜2.22s）← 破綻。T0が負＝定速区間ですら接触が切れている
```

```
幅比: HaR 1.54倍   HaL 0.82倍  ← 1未満！速度を落としているのに接触時間が短い
```

```
左右差（HaR面積/HaL面積）:
  刃元 1.40 → 中央 4.56 → 刃先 6.29（X=182mmで14.21のピーク後に低下）
```

**刃元はほぼ対称なのに刃先に向かって単調に開く = 平行ずれではなく取り付け角度誤差の特徴。**

推定角度誤差:
```
δ_actual(HaL刃先) = 2.5N ÷ 2.2 N/mm ≈ 1.1mm
ズレ = 3mm(名目) − 1.1mm ≈ 1.9mm
atan(1.9 / 174) ≈ 0.63°
```

**わずか0.6°の傾きでHaLの力分布が壊れている。** J6で180°反転するため、傾きが符号を変えて効く。

※ HaLの刃先4本の面積が `3.27/3.29/3.27/3.32` とほぼ一定＝**ノイズフロアに張り付いている**。
　 X=182mmで左右比がピークを打って下がるのはHaL側が底打ちしているためで、**真の角度誤差はこの推定より大きい可能性が高い**。

### 3.4 力の向きが刃元と刃先で49.5度違う（たわみの直接証拠）

センサ座標系（`AXIS_MAP_FORCE` 適用前の生値）での平均力の向き:
```
刃元(#1) : [-0.498, -0.795, +0.347]
刃先(#19): [+0.260, -0.680, +0.686]   なす角 49.5度
```

センサはツールに固定なので幾何学的には一定であるべき。成分比を見ると：

| | fy（法線寄り） | fz（接線寄り） | 比 |
|---|---|---|---|
| 刃元 | −11.9 | +4.9 | 2.4 |
| 刃先 | −2.6 | +2.7 | 1.0 |

摩擦係数の違いで説明すると μ が 0.41→1.04 と2.5倍変わることになり不可能。
**解釈: 刃元は剛性が高く押した方向に力が立つ。刃先はたわんで砥石に沿って逃げる。**

### 3.5 速度指令は正しく効いている（オーバーライドは問題なし）

```
実測の山の周期:    5.21 → 6.94 s  （差 +1.72）
理論ストローク時間: 1.00 → 2.72 s  （差 +1.72）  ← 完全一致
周期 − 理論 = 4.21 s（全19本でピタリ一定）
```

実測が理論の約2.9倍かかるのは早送り＋接近/後退の固定費（4.21秒）。**速度変調とは無関係。**

### 3.6 押し込み方向

センサ座標系で **fy の負方向が主成分**（全19ストロークとも）。
ただし剛性の左右比較には `|F|` の傾きで十分：

```
合力 = 法線力 × √(1+μ²)
k_apparent = ΔF/Δδ = k_true × √(1+μ²)
```
μ が左右で同じなら √(1+μ²) は kR/kL で約分されて消える。**μ の絶対値は不要。**
（ただし k の絶対値を機種展開に使う段では √(1+μ²) の分離が必要。今は保留）

## 4. 実装タスク（Claude Code向け）

### 4.1 方針

`plot_force_log.py` に**既に以下がある**ので、再実装しないこと：

| 既存関数 | 役割 |
|---|---|
| `load_and_validate(path)` | CSV読み込み（dict of lists を返す。pandasではない） |
| `_find_air_csv(dirs)` | air.csv 探索 |
| `estimate_baseline_shift(d, base, max_lag, step)` | Pearson相関でラグ推定 → `(s, corr)` |
| `apply_baseline_persides(d, base)` | **面ごとに個別整列して減算**（全体整列より堅牢） |
| `detect_segments(t, fmag, thr, min_dur, merge_gap)` | 山検出 → `[(a,b), ...]` |
| `detect_phase_split(t, F, lo_frac, hi_frac, win_s)` | HaR/HaL境界の時刻 |
| `analyze_segments(d, thr)` | 区間統計（平均・最大・平均Fy・平均Mx）※**面積は無い** |
| `side_summary(d, split, right_first)` | 左右サマリ ※**面全体の統計でストローク単位ではない** |
| `auto_zero(d, active_thr, split, ref_frac)` | air.csvなしの自動ゼロ |

**既存に無く、新規に追加すべきもの（これだけ）**:

1. `stroke_areas()` — 区間 → 面積 `∫F dt` の算出とX座標の紐付け
2. `asymmetry()` — **ストローク単位**の左右比較（角度誤差診断）
3. `estimate_T0()` — 接触時間の下駄の推定
4. `suggest_speed()` — 面積を揃える速度カーブ逆算
5. `stiffness_sweep()` — 押し込みスイープ → 等価剛性 k(x)

**依存について**: `plot_force_log.py` は `_interp` / `_pearson` / `_median` を自前実装し numpy を使っていない。
この方針を維持するなら追加分も純Pythonで書く（最小二乗も20行程度で自前実装可）。
※ matplotlib が numpy に依存するため numpy 自体は環境に必ず存在する。方針判断は camellia さんに確認すること。

### 4.3 コアロジック（動作検証済み）

**面積の算出**（台形則）:
```python
def _trapz(y, x):
    """台形則の積分。numpyを使わない版。"""
    s = 0.0
    for i in range(len(x) - 1):
        s += (y[i] + y[i + 1]) * 0.5 * (x[i + 1] - x[i])
    return s

# 各ストロークについて
area = _trapz(F[a:b+1], t[a:b+1])     # [N·s] ← 除去量の代理指標
dur  = t[b] - t[a]
favg = area / dur
```

**T0の推定**:
```python
def estimate_T0(durations, speeds, stroke_dist=STROKE_DIST_MM):
    """接触時間の下駄 T0 を推定する。
    w(x) = stroke_dist/v(x) + T0 なので、w - stroke_dist/v の中央値を取る。
    HaRでは 2.50s（ばらつき±0.05s）と極めて安定。
    HaLでばらつきが大きい(-1.38〜2.22s)場合は接触が不安定＝位置ずれの疑い。
    """
    resid = [w - stroke_dist / v for w, v in zip(durations, speeds)]
    resid_sorted = sorted(resid)
    n = len(resid_sorted)
    med = resid_sorted[n // 2] if n % 2 else (resid_sorted[n//2-1] + resid_sorted[n//2]) * 0.5
    return med, min(resid), max(resid)   # 中央値とばらつきを返す（ばらつきが健全性の指標）
```

**速度カーブの逆算**:
```python
def suggest_speed(areas, durs, favgs, speeds, T0, stroke_dist=STROKE_DIST_MM):
    """面積を揃える送り速度カーブを逆算する。

    モデル: A(x) = Favg(x) · [stroke_dist/v(x) + T0]
    Favg は押し込み量で決まり送り速度に依存しない、と仮定（厳密でないため反復が必要）。

    T0>0 のため各ストロークの面積には下限 A_floor = Favg·T0 がある。
    全ストロークが到達可能な目標は max(A_floor) 以上でなければならない。
    """
    A_floor = [f * T0 for f in favgs]
    A_target = max(A_floor) * 1.02      # 到達可能な最小の目標値

    v_new = []
    for f in favgs:
        need = A_target / f - T0        # 必要な定速区間の時間
        if need <= 0:
            v_new.append(SPEED_MAX)
        else:
            v = stroke_dist / need
            v_new.append(min(max(v, SPEED_MIN), SPEED_MAX))
    return v_new, A_target, A_floor
```

**剛性同定（押し込みスイープ）**:
```python
def _lstsq_slope(xs, ys):
    """最小二乗で傾きと切片とR²を返す（numpy不要）。"""
    n = len(xs)
    mx = sum(xs) / n
    my = sum(ys) / n
    sxx = sum((x - mx) ** 2 for x in xs)
    sxy = sum((x - mx) * (y - my) for x, y in zip(xs, ys))
    if sxx == 0:
        return 0.0, my, 0.0
    k = sxy / sxx
    b = my - k * mx
    ss_tot = sum((y - my) ** 2 for y in ys)
    ss_res = sum((y - (k * x + b)) ** 2 for x, y in zip(xs, ys))
    r2 = 1 - ss_res / ss_tot if ss_tot > 0 else float('nan')
    return k, b, r2

def stiffness_sweep(pushes, favgs_per_push):
    """押し込みスイープから等価剛性 k(x) = ΔF/Δδ を同定する。

    pushes: [2.0, 2.5, 3.0, 3.5, 4.0]  押し込み量 [mm]
    favgs_per_push: [[19本のfavg], [19本のfavg], ...]  各押し込み量での平均力

    ★傾きを使うので δ の絶対値も接触開始点の誤差も影響しない。
      これが「E や L_eff を仮定せずに剛性が測れる」理由。
    """
    n_st = min(len(f) for f in favgs_per_push)
    ks, r2s = [], []
    for si in range(n_st):
        F = [f[si] for f in favgs_per_push]
        k, b, r2 = _lstsq_slope(pushes, F)
        ks.append(k)
        r2s.append(r2)
    return ks, r2s    # k [N/mm], R²（R²<0.9 なら直線性が悪い＝要確認）
```

**刻み幅が0.5mmである理由**（当初0.1mmと言っていたが訂正済み）:

## 8. 落とし穴（ハマりどころ）

1. **air.csv を引かずに解析しない** — 2面目の力が過小評価される（姿勢オフセット8.6N）
2. **ピーク値で速度を決めない** — 面積（∫F dt）でないと接触時間の情報が落ちる
3. **面積比 ÷ 速度比 は二重補正** — 面積は既に「高さ×幅」で幅が速度の逆数を含む。ここで一度間違えた
4. **面分割を固定秒数でやらない** — 面内ギャップが速度変調で1.7〜4sと幅がある。ギャップ中央値×3 か既存の `detect_phase_split` を使う
5. **T0 を無視して v ∝ 面積 で逆算しない** — 収束しない。刃元には `A_floor = Favg·T0` の下限がある
6. **位置がずれたまま速度カーブを校正しない** — セッティング依存の歪んだカーブになり、包丁を付け直した瞬間に破綻する
